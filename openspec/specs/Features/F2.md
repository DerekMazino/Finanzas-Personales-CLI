### Feature: Gestion de Ingresos y Egresos
## HU1: Agregar Concepto
# Descripción
- **Como** Usuario 
- **Requiero** Agregar un nuevo concepto que puede ser un ingreso o egreso.
- **Para así** Poder gestionar mis conceptos del mes.
# Criterios
- **Criterio 1** Dado que el usuario seleccionó la opción para ingresar un nuevo concepto, el usuario debe indicar el nombre, valor, tipo (ingreso o egreso), periodo al que pertenece y si es un concepto recurrente.
- **Criterio 2** Validaciones de datos:
    - El **valor** debe ser un número positivo.
    - El **nombre** debe comenzar con una letra del alfabeto (a-z, A-Z).
    - El **tipo** debe ser estrictamente 'ingreso' o 'egreso'.
- **Criterio 3** Si el periodo seleccionado para el concepto no existe:
    - El sistema debe preguntar: "¿El periodo [Mes/Año] no existe, desea crearlo? [S/N]".
    - Si el usuario acepta, se crea el periodo y luego se registra el concepto.
    - Si el usuario rechaza, se cancela la operación con un mensaje informativo.
- **Criterio 4** Si el concepto se marca como **Recurrente**:
    - El sistema debe guardarlo como una plantilla que se replicará automáticamente cada vez que se cree un nuevo periodo futuro.
    - En el nuevo periodo, el valor del concepto recurrente podrá ser modificado sin afectar a otros periodos.

## HU2: Listar Conceptos
# Descripción
- **Como** Usuario 
- **Requiero** Listar todos los conceptos registrados en un periodo específico.
- **Para así** Poder ver el detalle de mis finanzas.
# Criterios
- **Criterio 1** Dado que el usuario visualiza un periodo, el sistema debe mostrar una tabla con: Nombre, Valor, Tipo y si es Recurrente.
- **Criterio 2** Si no hay conceptos en el periodo seleccionado, el sistema debe mostrar un mensaje: "No hay conceptos registrados para este periodo. ¡Comienza agregando uno!".
- **Criterio 3** Si el concepto se crea correctamente, debe almacenarse la fecha de creación. El valor por defecto sera la fecha actual.
- **Criterio 4** Si el concepto se marca como **Recurrente**:
    - El sistema debe guardarlo como una plantilla que se replicará automáticamente cada vez que se cree un nuevo periodo futuro.
    - Los periodos anteriores a la fecha actual no deben verse afectados por cambios en los conceptos recurrentes.

## HU3: Modificar Conceptos
# Descripción
- **Como** Usuario 
- **Requiero** Modificar un concepto registrado en un periodo específico.
- **Para así** Poder corregir errores o actualizar información.
# Criterios
- **Criterio 1** El usuario debe indicar el **ID** del concepto a modificar (obtenido del listado).
- **Criterio 2** Validaciones de datos:
    - El **valor** debe ser un número positivo.
    - El **nombre** (si se cambia) debe comenzar con una letra (a-z, A-Z).
- **Criterio 3** **Lógica de cambios en recurrentes**:
    - Si se modifica el **valor** o **tipo**, el cambio afecta solo al periodo actual.
    - Si se modifica el **nombre**, el sistema actualizará la plantilla recurrente para futuros periodos. Los periodos pasados conservan el nombre original.

## HU4: Eliminar Conceptos
# Descripción
- **Como** Usuario 
- **Requiero** Eliminar un concepto registrado.
- **Para así** Poder corregir errores.
# Criterios
- **Criterio 1** El usuario debe indicar el **ID** del concepto a eliminar.
- **Criterio 2** **Lógica de eliminación inteligente**:
    - **Física**: Si el concepto no tiene registros en ningún otro periodo, se elimina definitivamente de la base de datos.
    - **Lógica**: Si el concepto tiene historial en otros periodos, se marca como "Inactivo" en la plantilla de recurrencia. Esto evita que aparezca en el futuro pero preserva el historial pasado.
- **Criterio 3** El sistema debe confirmar la acción antes de proceder, informando si será una eliminación definitiva o solo una desactivación de recurrencia.
